# 🇹🇳 Tunisian Dialect LLM — Full Training Pipeline

A step-by-step technical guide for building a low-resource dialect language model from scratch.

---

## Overview

Tunisian Arabic (Darija) is one of the hardest dialects for current LLMs — even GPT-4 struggles with it. The reasons are straightforward: virtually no Tunisian text exists in standard pre-training corpora, and the dialect's code-switching behavior (Arabic/French/Arabizi in the same sentence) is unlike anything models have seen at scale.

This pipeline provides a complete, practical approach to building a Tunisian dialect LLM using modern parameter-efficient training techniques, achievable with modest compute resources.

---

## Pipeline at a Glance

| Stage | Name | Goal | Data Type |
|-------|------|------|-----------|
| 1 | **Data Collection** | Gather raw Tunisian text from diverse sources | Unlabeled text |
| 2 | **Continued Pre-training** | Teach the base LLM Tunisian vocabulary & patterns | Raw text (no labels) |
| 3 | **Lexicon RAG Setup** | Ground the model in Tunisian-specific expressions | Glossaries / books |
| 4 | **Synthetic Q/A Generation** | Create instruction-following training data | Q/A pairs (autogenerated) |
| 5 | **Supervised Fine-Tuning (SFT)** | Train the model to answer in Tunisian dialect | Curated Q/A pairs |

---

## Stage 1 — Data Collection

Gather raw Tunisian text from the real world.

**Good data should be:**
- Written in how Tunisians actually write (not MSA, not classical Arabic)
- Diverse in register: informal conversations, commentary, humor, news discussion
- From multiple domains: everyday life, food, culture, politics, sport

### Recommended Sources

**Existing Academic Corpora**
- [Tunisiya Corpus](https://tunisiya.org) — 1.18M words, digitized novels and SMS/Facebook messages
- **STAC Corpus** — 5 hours of transcribed spontaneous Tunisian Arabic speech with morphosyntactic annotations
- **T-HSAB** — Tunisian hate speech/abusive language dataset (also useful as general dialect text)
- **TunBERT Common Crawl data** — community-shared pre-training data from HuggingFace

**Web Scraping**
- Twitter/X — filter by Tunisia geolocation bounding box or common Tunisian hashtags
- Facebook public groups — cooking groups, humor pages, local news comments
- Tunisian forums — Tunisianet and similar local forums (years of archived content)
- YouTube subtitles — Tunisian YouTubers, TV show clips, comedy sketches
- Reddit r/Tunisia — mostly Arabizi/French but still valuable

**Books & Glossaries**
- Books explaining Tunisian expressions in Arabic or French
- Tunisian proverb collections
- Dialect dictionaries (even partial ones are useful)

### Data Cleaning Steps

1. Remove spam, ads, and bot-generated content
2. Filter out text that is almost entirely MSA or French with no Tunisian dialect
3. Deduplicate using MinHash or similar near-duplicate detection
4. Normalize inconsistent whitespace, URLs, and special characters
5. **Do NOT normalize spelling variation** — that variation IS the dialect

**Target size:** 50–100 million tokens (~300–600 MB of text). Quality matters more than volume.

---

## Stage 2 — Continued Pre-training

Teach the base model Tunisian text patterns from raw unlabeled data.

This is *domain-adaptive pre-training*: take an existing pre-trained LLM and continue training it on your Tunisian corpus using next-token prediction. No labels, no Q/A structure — just raw text.

**Why this step matters:** Without it, SFT forces the model to simultaneously learn Tunisian vocabulary, response structure, *and* grammar at once. Separating the "learn the language" step from the "learn to answer" step makes both stages more data-efficient.

### Training Setup

**Data format:** Concatenate cleaned text into long chunks packed into 2048 or 4096 token blocks, separated by an end-of-document token.

**Use QLoRA** (Quantized Low-Rank Adaptation):
- Quantize the base model to 4-bit to reduce memory footprint
- Add small trainable adapter layers (LoRA rank 16–64)
- Only adapter weights are updated — not the full model
- Achievable on a single 24 GB GPU or a free Colab A100

### Recommended Hyperparameters

| Parameter | Value |
|-----------|-------|
| Learning rate | `2e-5` to `5e-5` |
| Batch size | 4–8 with gradient accumulation |
| Epochs | 1–3 (more risks forgetting general knowledge) |
| Sequence length | 2048–4096 tokens |
| Scheduler | Cosine with warmup |

### Recommended Tools

| Tool | Purpose |
|------|---------|
| [Unsloth](https://github.com/unslothai/unsloth) | Fastest QLoRA training, free Colab notebooks |
| [HuggingFace TRL](https://github.com/huggingface/trl) | Flexible, well-documented SFTTrainer |
| [Axolotl](https://github.com/OpenAccess-AI-Collective/axolotl) | Configuration-based, good for reproducibility |

**Output:** A model that produces fluent Tunisian text — but not yet a conversational assistant. That comes in Stage 5.

## Stage 3 — Lexicon RAG Setup

Ground the model in Tunisian vocabulary using retrieval-augmented generation.

Instead of baking all vocabulary into model weights, RAG provides relevant information at inference time by retrieving it from an external knowledge base and injecting it into context.

**Why RAG for a dialect?**
- Add new vocabulary entries without retraining
- Ground answers in authoritative dialect reference material
- Handle regional variations
- Cheaply update the knowledge base as new sources are found

### Building the Knowledge Base

**What to include:**
- Tunisian expression books (proverbs, slang)
- Dialect dictionaries
- Cultural concept explanations (foods, traditions, places)

**Entry structure:**
```json
{
  "term": "barcha",
  "meaning_ar": "كثير",
  "meaning_fr": "beaucoup, très",
  "example": "3andi barcha khobza = I have a lot of bread"
}
```

### Embedding & Indexing

1. Convert each entry to text
2. Embed with `multilingual-e5-large` or `paraphrase-multilingual-mpnet`
3. Store in a vector database: **ChromaDB** (local/simple), **Qdrant** (scalable), or **FAISS** (lightweight)

### Retrieval at Inference Time

1. Embed the user's query
2. Retrieve top 3–5 most semantically similar glossary entries
3. Prepend them to the system prompt
4. Pass the enriched prompt to the LLM

> **RAG vs Fine-tuning:** RAG provides specific factual grounding (*what does this expression mean?*). Fine-tuning teaches conversational behavior (*how should I answer in Tunisian?*). Use both together.

## Stage 4 — Synthetic Q/A Generation

Use a teacher LLM to generate the instruction-following dataset you cannot collect manually.

Collecting real Tunisian Q/A pairs manually is slow and hard to scale. Instead, use a powerful general-purpose LLM (the "teacher") to generate a large synthetic dataset, seeded from a small set of human-written examples — the same approach used to build Alpaca, WizardLM, and the Tunisian Arabic benchmark dataset.

### The Self-Instruct Approach

1. Write 10–20 high-quality Tunisian Q/A pairs by hand (seed examples)
2. Give them to the teacher LLM with a generation prompt
3. Filter and clean the outputs
4. Repeat, using new examples as additional seeds
5. Scale to hundreds or thousands of pairs

### Prompting the Teacher LLM

```
System: You are a native Tunisian Arabic speaker.
Generate realistic Q&A pairs in Tunisian dialect.
Use natural Darija — mix Arabic script and French loanwords
as a real speaker would.
```

### Filtering & Validation

- Remove pairs where the answer is in MSA or French only
- Remove duplicates or near-duplicates
- Remove logically mismatched question/answer pairs
- Have native speakers review a random 5–10% sample
- Flag and remove culturally inaccurate content

**Target size:** 2,000–10,000 curated Q/A pairs. Research shows that even a few hundred high-quality examples dramatically improves instruction-following over zero-shot.

---

## Stage 5 — Supervised Fine-Tuning (SFT)

Train the dialect-aware model to follow instructions and answer in Tunisian.

Take the Stage 2 model (already familiar with Tunisian text) and teach it to behave like a conversational assistant using your curated Q/A pairs formatted as instruction-response examples.

### Training Configuration

| Hyperparameter | Recommended Value | Notes |
|----------------|-------------------|-------|
| Learning rate | `1e-4` to `2e-4` | Higher than pre-training; SFT data is small |
| Batch size | 4–8 + gradient accumulation | Effective batch 32–64 is ideal |
| LoRA rank | 16–64 | Higher rank = more capacity = more memory |
| Epochs | 2–5 | Watch for overfitting on small datasets |
| Max sequence length | 512–1024 tokens | Q/A pairs are shorter than pre-training docs |
| Scheduler | Cosine with warmup | 10% warmup steps recommended |

> **Important:** Compute the loss only on response tokens, not instruction tokens. Most SFT frameworks (TRL's SFTTrainer, Unsloth) handle this automatically.

### Iterative Improvement Loop

1. Train on initial synthetic Q/A dataset
2. Run the model on held-out Tunisian questions
3. Have native speakers evaluate responses (1–5 on accuracy, fluency, dialect authenticity)
4. Identify failure categories (wrong dialect, factual error, too formal)
5. Generate more synthetic data targeting those failure categories
6. Repeat

### Preventing Catastrophic Forgetting

Mix 5–10% of general Arabic or multilingual instruction data into your SFT dataset. This preserves the model's general reasoning ability while adding dialect expertise.

---

## Evaluation

### Human Evaluation (Primary)
- Native Tunisian speakers rate responses on: dialect authenticity (1–5), correctness (1–5), fluency (1–5)
- Use 50–100 held-out questions covering different topics

### LLM-as-Judge
- Use GPT-4 as an evaluator (as done in the Tunisian Arabic benchmark paper)
- Compare model answers against reference answers for quality, correctness, and Tunisian usage
- Cheaper than human evaluation for frequent iteration

### Task-Specific Benchmarks
- Classification tasks: TunBERT evaluation sets for sentiment analysis or dialect ID
- Generation: prefer human scores or LLM-judge over BLEU/ROUGE (too sensitive to spelling variation)

## Known Risks & Mitigations

| Risk | What Happens | Mitigation |
|------|-------------|------------|
| Hallucination from synthetic data | Model learns wrong facts from bad teacher outputs | Human validation; filter before training |
| Script inconsistency | Works in Arabic script but not Arabizi (or vice versa) | Ensure both scripts are represented at every stage |
| Catastrophic forgetting | Model loses general ability after SFT | Mix 5–10% general data into SFT; use low learning rate |
| Overfitting on small SFT data | Model memorizes pairs instead of generalizing | Use early stopping; watch validation loss |
| Regional dialect bias | Model trained on Tunis dialect only | Collect data from multiple regions |
| Teacher model errors | GPT-4 generates MSA instead of Darija | Strong system prompt; few-shot examples; filter outputs |

## Quick Start (Weekend Baseline)

```
1. Download the Tunisiya corpus from tunisiya.org
2. Download TunBERT's pre-training data from HuggingFace
3. Write 20 seed Q/A pairs by hand
4. Generate 500 synthetic pairs using GPT-4
5. Fine-tune Llama-3-8B with QLoRA using Unsloth on Colab
```

A 7B base model + 50 MB of dialect text + 1,000 synthetic Q/A pairs + a glossary of 500 expressions will already outperform any general-purpose LLM on Tunisian dialect today.

---

## Notes

This pipeline is a cycle, not a one-shot process. Each evaluation round reveals gaps that drive new data collection, new synthetic generation, and new fine-tuning runs. The most important early investment is **data quality** — specifically the seed Q/A examples for Stage 4 and the lexicon knowledge base for Stage 3.
